# 产品需求文档

## 产品目标

建设一套“单站点、单导航、页面级登录”的文档平台，让多个项目在保留源仓 `docs/` 自治的前提下，统一发布到同一站点。

## 功能需求

| ID | 需求 | 验收标准 |
| --- | --- | --- |
| REQ-001 | 源仓必须通过统一 `docs/` 标准编写文档。 | 每个目录都有 `index.md`，导航和权限都来自 `index.md` 元数据。 |
| REQ-002 | 系统必须支持从多个源仓获取文档。 | 能基于仓库配置拉取多个文档源。 |
| REQ-003 | 外部源仓同步时只获取 `docs/` 目录。 | Git 同步命令使用 submodule 更新，并在子仓内只对 `docs` 执行 sparse-checkout。 |
| REQ-004 | 系统必须在静态构建前或后生成页面权限清单。 | 产出 `permissions.json` 和 Nginx 私有页面规则文件。 |
| REQ-005 | 系统必须支持单站点展示全部页面，并对私有页面触发登录。 | 私有页面路径被 Nginx 拦截并重定向到认证入口。 |
| REQ-006 | 认证链路必须支持用户名密码和 GitHub 登录。 | 部署方案包含 Casdoor 和 GitHub Provider 配置。 |
| REQ-007 | 宿主机 Nginx 必须同时承接静态页面与认证代理入口。 | 宿主机 `Nginx` 配置承接文档域名与认证域名，Docker Compose 仅运行认证相关服务。 |
| REQ-008 | 系统必须保留可追溯的需求、方案、部署和运维文档。 | `docs/` 中存在对应阶段文档。 |

## 非功能需求

| ID | 要求 |
| --- | --- |
| NFR-001 | 根仓不得改写源仓页面内容和文件名。 |
| NFR-002 | 权限规则必须与页面路径一一对应，可审计。 |
| NFR-003 | 同步、构建、权限生成应可重复执行。 |
| NFR-004 | 未列入 `index.md` 元数据的页面必须视为构建错误。 |
| NFR-005 | 私有页面的访问控制必须在网关层生效，而不是只在前端提示。 |
