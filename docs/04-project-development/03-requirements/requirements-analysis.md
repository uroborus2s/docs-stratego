# 需求分析

## 核心对象

- 源仓文档作者：维护 `docs/` 内容、目录顺序和访问级别
- 平台管理员：同步源仓、生成站点、部署认证链路
- 匿名读者：阅读公开页面
- 已登录用户：阅读私有页面
- 根仓维护者：审核共享 bot PR，决定何时把子仓最新文档版本并入 `main`

## 本次变更范围

- `CR-001` 聚焦“子仓更新自动通知 + 根仓共享 bot PR + 人工合并后再发布”。
- `CR-002` 聚焦“把公开接入事实源从内部设计层迁到 Contributor Guide，并按读者任务重构阅读路径”。
- `CR-003` 聚焦“提供接入/移除辅助 CLI，并把命令说明并入使用指南”。

## 关键分析

### 为什么用目录 `index.md` 维护权限

- 整个项目的页面权限和目录树收敛到一个根清单文件，维护边界更清晰
- 导航结构和访问级别在同一个地方声明，避免多层目录各自维护时漂移
- 平台构建器可以校验“文件系统里是否有漏声明页面”

### 为什么要单独生成权限清单

- 静态站点本身不具备运行时访问控制
- Nginx 需要一份明确的 URL 规则清单
- 权限清单还能作为审计产物，方便检查私有页面是否配置正确

### 为什么选择 Git submodule + sparse-checkout

- 目标不是同步业务仓全部内容，而是只拿 `docs/`
- submodule 让根仓能显式登记外部项目来源
- sparse-checkout 让子仓工作区只展开 `docs/`

### 为什么要把“同步子仓指针”和“正式发布”拆开

- 子仓推送文档后立即发布会绕过根仓审核，和当前 PR 闭环治理冲突
- 子仓文档可能会破坏聚合构建，先停留在共享 bot PR 中更容易排障
- 同步指针与正式发布是两类不同风险：前者关注版本收口，后者关注站点交付

### 为什么由子仓主动发送 `repository_dispatch`

- 根仓不需要轮询所有源仓，事件驱动更直接
- 子仓可通过 `paths: docs/**` 把触发范围限制在文档变更
- 接入新子仓时，只需补一个轻量 workflow 和 Secret，运维成本更低

### 为什么复用一个共享 bot PR

- 同一时间只保留一个“待同步指针”的审核入口，减少 PR 噪音
- 多个子仓连续更新时，reviewer 只需看一份 diff 和一组 checks
- 如果某个子仓阻塞构建，其余待同步指针仍可在同一个 PR 中集中处理

### 为什么要把子仓新增文件写入用户指南

- 当前接入流程已经依赖 `config/source-repos.json`、`.gitmodules` 和 GitHub App；新增自动联动后，子仓还需要固定的 workflow 文件
- 如果不把“子仓必须新增 `.github/workflows/notify-docs-stratego.yml`”写成正式事实源，后续接入很容易只改根仓而漏改子仓
- 文档要直接回答“子仓需要加哪些文件、哪些 Secret”，否则接入动作不可重复

### 为什么要把源文档标准公开化并收口到 Contributor Guide

- 接入方首先需要的是任务导向的说明，而不是内部设计推导过程
- `usage.md` 和私有设计文档过去重复解释同一套规则，阅读和维护都不舒服
- 按“标准 -> 接入 -> 联动 -> 移除 -> CLI”组织更贴近真实操作顺序

### 为什么要补正式 CLI，而不是继续依赖分散脚本

- 接入和移除动作跨越“源仓本地”和“根仓本地”两侧，单个脚本名很难表达上下文
- `source add/remove/validate/scaffold-notify` 这种命名更接近读者心智模型
- 公开文档可以直接围绕命令分组，减少“看文档时还要自行拼脚本参数”的成本

## 风险

- 如果私有页面和公开页面共用静态资源目录，可能出现资源保护不清晰
- 如果开启匿名搜索索引，可能暴露私有页面标题或摘要
- 如果源仓不严格维护 `index.md`，导航和权限会直接失真
- 如果根仓写入 bot 分支的凭证范围过大，可能引入额外安全面
- 如果多个子仓同时推送，缺少并发收口会让共享 PR 抖动或相互覆盖
- 如果共享 bot PR 长时间未合并，线上版本会落后于子仓实际最新文档
- 如果 CLI 默认执行删除或写入，会提高误操作风险，因此必须提供 `--dry-run`、`--yes` 等显式开关
