# 需求分析

## 核心对象

- 源仓文档作者：维护 `docs/` 内容、目录顺序和访问级别
- 平台管理员：同步源仓、生成站点、部署认证链路
- 匿名读者：阅读公开页面
- 已登录用户：阅读私有页面

## 关键分析

### 为什么用目录 `index.md` 维护权限

- 整个项目的页面权限和目录树收敛到一个根清单文件，维护边界更清晰
- 导航结构和访问级别在同一个地方声明，避免多层目录各自维护时漂移
- 平台构建器可以校验“文件系统里是否有漏声明页面”

### 为什么要单独生成权限清单

- 静态站点本身不具备运行时访问控制
- Nginx 需要一份明确的 URL 规则清单
- 权限清单还能作为审计产物，方便检查私有页面是否配置正确

### 为什么选择 Git submodule + sparse-checkout

- 目标不是同步业务仓全部内容，而是只拿 `docs/`
- submodule 让根仓能显式登记外部项目来源
- sparse-checkout 让子仓工作区只展开 `docs/`

## 风险

- 如果私有页面和公开页面共用静态资源目录，可能出现资源保护不清晰
- 如果开启匿名搜索索引，可能暴露私有页面标题或摘要
- 如果源仓不严格维护 `index.md`，导航和权限会直接失真
