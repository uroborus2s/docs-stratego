# 服务器部署 SOP

## 1. 目标

本文是面向运维执行人的逐步操作手册，用来把 `docs-stratego` 首次落到云服务器，并启用后续 GitHub 自动发布。

## 2. 服务器前置条件

- 已安装 Git
- 已安装 Docker 与 `docker compose`
- 宿主机已安装 `Nginx`
- 宿主机可访问 GitHub
- Docker 内已有 Redis
- 已准备文档域名与认证域名

## 3. 首次引导

### 3.1 稀疏拉取认证运行目录

```bash
git clone --filter=blob:none --no-checkout https://github.com/uroborus2s/stratego-docs.git ~/docs-stratego
cd ~/docs-stratego
git sparse-checkout init --no-cone
git sparse-checkout set /deploy/docker-compose.yml /deploy/casdoor/ /deploy/casdoor/** /deploy/oauth2-proxy/ /deploy/oauth2-proxy/**
git checkout main
mkdir -p ~/docs-stratego/deploy/casdoor/data
sudo install -d -m 0755 -o root -g root /var/www/docs-stratego
sudo install -d -m 0755 -o root -g root /etc/nginx/snippets/docs-stratego
printf "# generated by docs-stratego deploy\n" | sudo tee /etc/nginx/snippets/docs-stratego/private_locations.conf >/dev/null
sudo chmod 0644 /etc/nginx/snippets/docs-stratego/private_locations.conf
```

### 3.2 准备 Docker 网络

```bash
docker network create docs-auth-internal
```

然后确认现有 Redis 保持在原业务网络里，例如 `webapp_wps_net`。  
推荐做法不是迁移 Redis，而是让 `oauth2-proxy` 同时加入：

- `docs-auth-internal`
- `webapp_wps_net`

如果 Redis 容器名不是 `redis`，请同步修改 `deploy/oauth2-proxy/oauth2-proxy.cfg` 里的 `redis_connection_url`。

### 3.3 编辑运行时配置

编辑：

- `deploy/casdoor/app.conf`
- `deploy/oauth2-proxy/oauth2-proxy.cfg`

至少确认：

- Casdoor SQLite 路径
- `oidc_issuer_url`
- `redirect_url`
- `client_id`
- `client_secret`
- `cookie_secret`
- `redis_connection_url`

## 4. 首次发布

```bash
cd ~/docs-stratego
export DOCS_INTERNAL_DOCKER_NETWORK=docs-auth-internal
export DOCS_REDIS_DOCKER_NETWORK=webapp_wps_net
docker compose -f deploy/docker-compose.yml up -d
```

如果是首次上线，先由运维手工安装宿主机 HTTP 引导配置：

```bash
sudo ln -sf /etc/nginx/sites-available/docs.example.com /etc/nginx/sites-enabled/docs.example.com
sudo ln -sf /etc/nginx/sites-available/auth.docs.example.com /etc/nginx/sites-enabled/auth.docs.example.com
sudo nginx -t
sudo systemctl reload nginx
```

然后触发一次 GitHub Actions 发布，让 Runner 把第一版 `site/` 和 `private_locations.conf` 安装到服务器。证书、GitHub App 和正式 HTTPS 配置的完整步骤，统一按 [云服务器部署与 CI/CD 实操](../../02-user-guide/cloud-server-cicd-playbook.md) 执行。

## 5. 上线检查

### 5.1 服务状态

```bash
docker compose -f deploy/docker-compose.yml ps
sudo nginx -t
```

### 5.2 页面访问

- 打开首页
- 打开一个公开页面
- 打开一个私有页面
- 确认跳转登录
- 登录后返回原页面

### 5.3 配置文件落地

确认存在：

- `/etc/nginx/sites-available/docs.example.com`
- `/etc/nginx/sites-available/auth.docs.example.com`
- `/etc/nginx/snippets/docs-stratego/private_locations.conf`

## 6. 启用 GitHub 自动发布

在 GitHub 仓库中配置这些 Secrets：

- `DOCS_DEPLOY_HOST`
- `DOCS_DEPLOY_USER`
- `DOCS_DEPLOY_SSH_KEY`
- `DOCS_DEPLOY_PORT`
- `DOCS_DEPLOY_SITE_DIR`
- `DOCS_PRIVATE_LOCATIONS_PATH`
- `DOCS_RELOAD_HOST_NGINX`

然后向 `main` 提交一次空变更，验证 workflow 能否跑通。

## 7. 日常发布

后续只需要：

1. 提交变更到 `main/master`
2. 等待 GitHub Actions 执行
3. 在服务器上抽查 `/var/www/docs-stratego`、`private_locations.conf` 是否已更新
4. 如需排障，下载保留 7 天的 Actions artifact 查看 `site/` 与 `private_locations.conf`
5. 在浏览器中抽查首页和一个私有页面

## 8. 回滚

```bash
cd ~/docs-stratego-full
git log --oneline -n 10
git checkout <stable-tag-or-commit>
export SKIP_GIT_PULL=1
bash scripts/deploy_remote.sh
```

如果自动发布可用，优先重新发布上一个稳定 commit/tag 对应的制品，而不是直接在服务器上做全量回退。

## 9. 应急处理

### oauth2-proxy 起不来

- 检查 `oauth2-proxy` 是否已接入 Redis 所在网络
- 检查 `client_id/client_secret`

### Casdoor 起不来

- 检查 SQLite 文件权限
- 检查 `app.conf`

### Nginx reload 失败

- 先执行 `sudo nginx -t`
- 修复运维安装的站点配置或路径后再 reload
